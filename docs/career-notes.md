# AgentFlow 简历与面试材料

## 一句话项目定位

独立设计并实现面向企业工单自动化的 AI Agent 执行与评测平台，支持结构化规划、受约束 Tool Calling、高风险人工审批、SSE 实时 Trace、会话恢复和确定性回归评测。

## 简历项目描述

**AgentFlow｜企业级 AI Agent 执行与评测平台**  
技术栈：TypeScript、Vue 3、Fastify、SSE、Zod、OpenAI-compatible API、pnpm Monorepo

- 设计 Planner、Executor、Replanner 执行链路，按计划步骤授予最小工具权限，并通过 Tool Registry 统一管理8个业务工具的 Schema、风险等级和执行入口。
- 实现 Human-in-the-loop 审批状态机，高风险退款在写入前暂停；审批拒绝后阻止退款和后续状态更新，并通过自动化测试验证完整业务状态零副作用。
- 建设 Agent 可观测与恢复能力，通过 SSE 展示计划、工具调用、审批和错误 Trace，持久化 Run、会话与评测快照，并对服务重启后的未完成任务进行中断降级。
- 搭建18条 golden task 的确定性评测集，覆盖查询、知识检索、退款、审批边界、异常安全和幂等性，同时断言最终回复、工具轨迹与业务副作用。
- 使用 DeepSeek 进行两轮18条真实模型回归，两轮均通过14条；通过失败漂移定位模型非确定性，将工具选择、审批属性、规则关键词和退款写入资格下沉到服务端约束。
- 构建命令行评测与 A/B 报告工具，区分 Mock/真实模式，真实评测强制关闭 fallback，并输出耗时、Token、失败诊断及 recovered/regressed 结果。

## 面试开场讲法

> 企业 Agent 的难点不是让模型调用工具，而是保证模型不会越权、不会重复写入，高风险动作能够人工接管，并且模型或 Prompt 更新后可以发现能力回归。这个项目围绕这四个问题构建了受约束执行器、工具风险治理、审批状态机、全链路 Trace 和确定性评测平台。

## 三个重点设计

### 1. 为什么不是自由循环 Agent

模型每轮只能调用当前计划步骤授权的一项工具。Executor 校验工具名、计划授权和参数 Schema，成功后才推进计划游标；计划耗尽后由 Executor 主动收敛，避免模型重复调用最后一个工具。

### 2. 为什么审批不能只写在 Prompt 中

Prompt 只能影响模型行为，不能构成安全边界。项目根据 Tool Registry 的 `riskLevel` 决定是否暂停，高风险工具必须产生 ApprovalRequest；拒绝后不会执行工具，也不会推进依赖写入步骤。

### 3. 为什么采用确定性 Judge

退款等资金场景需要验证真实副作用。评测不仅判断回答文本，还检查必须/禁止工具、审批步骤、工具调用次数、工单状态、订单退款状态和退款记录数量。相比单独使用 LLM Judge，更适合验证强业务约束。

## 真实评测故事

第一次3条 DeepSeek 冒烟仅通过1条。Trace 显示最终总结丢失查询明细，Action Planner 因缺少业务日期无法判断30天退款窗口。补齐信息保真约束和固定业务时钟后，第二轮3/3通过。

第一次18条全量评测通过14条。针对4条失败修复执行器协议、计划收敛与状态机后，定向复测4/4通过；但第二次全量仍为14/18，出现3条恢复和3条新回归。这说明“修一条 Prompt”不能保证系统稳定，因此进一步将可确定的决策下沉到服务端。

这个结果不包装成全量100%，而用于说明如何通过 Trace 和回归数据发现模型非确定性，并将安全约束从概率模型迁移到确定性代码。

## 常见追问

### 为什么选择 SSE？

当前交互主要是服务端单向推送执行事件，审批决策通过独立 HTTP 请求提交。SSE 实现简单、自带事件流语义和浏览器支持；如果未来需要高频双向协作或多参与者实时通信，再考虑 WebSocket。

### 服务重启时审批如何恢复？

当前 Promise 无法跨进程恢复，所以重启后将 `running` / `waiting_approval` Run 降级为可审计、可重试的中断状态，并清理失效 pending approval。生产版会使用数据库状态机和持久化任务队列恢复。

### 如何防止重复退款？

业务工具会复用同一订单、同一状态的既有退款记录；订单处于 `pending_approval` 时，状态机阻止工单被关闭。生产版还会增加数据库唯一约束和幂等键。

### 当前项目最大边界是什么？

状态保存在本地 JSON，适合单机演示而非分布式生产。下一步应迁移 PostgreSQL、队列与事务边界，而不是继续增加更多页面或无业务必要的多 Agent。

## 不能夸大的表述

- 不说“真实模型18/18通过”；当前两轮完整评测均为14/18。
- 不把 Mock Token 估算描述为模型官方 usage。
- 不把本地 JSON 描述成生产级持久化。
- 不声称拥有真实企业用户或线上业务收益。
