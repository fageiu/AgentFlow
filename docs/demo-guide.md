# AgentFlow 3～5 分钟演示脚本

## 录制前准备

- 使用 Node.js 20，执行 `pnpm install` 和 `pnpm dev`。
- 浏览器打开 `http://127.0.0.1:5173`，确认后端健康检查正常。
- 重置沙箱，确保退款记录为空，T-1001 为 `open`，O-7001 为 `none`。
- 录制时隐藏 `.env`、终端环境变量和任何 API Key。
- 浏览器缩放调整到一屏能看到任务输入、执行时间线和业务状态。

## 0:00～0:30 项目定位

操作：展示工作台首页和 README 标题。

讲解：

> AgentFlow 是一个企业流程 Agent 执行与评测平台。它关注的不只是模型能否调用工具，而是如何约束执行、控制高风险写入、保留完整 Trace，并通过可重复评测发现能力回归。

## 0:30～1:10 只读查询

输入：

```text
查询高优先级工单有哪些。
```

展示重点：

- Planner 只授权 `searchTickets`。
- Tool Registry 将其识别为只读工具。
- 最终回复包含具体工单号。
- 沙箱退款和工单状态没有变化。

讲解：

> 查询条件会在服务端归一化为筛选工具，模型不能把只读任务升级为退款或状态写入。

## 1:10～2:20 高风险审批拒绝

输入：

```text
处理工单 T-1001：判断退款条件，必要时创建退款并更新工单状态。
```

操作：等待 `approval_required`，点击拒绝。

展示重点：

- Agent 依次读取工单、客户、订单和退款规则。
- `createRefund` 风险等级为 `high`，执行流暂停。
- 拒绝后没有退款记录。
- O-7001 保持 `none`，T-1001 保持 `open`。

讲解：

> 高风险门禁由服务端风险等级决定，不依赖 Prompt。审批拒绝会作为 tool message 回传模型，同时 Executor 终止后续状态写入，保证零副作用。

## 2:20～3:10 审批通过与幂等

再次执行相同任务，这次批准。

展示重点：

- 产生一条 `pending_approval` 退款记录。
- 工单进入 `waiting_approval`。
- 再执行一次相同任务，退款记录仍只有一条。

讲解：

> 写工具具备业务幂等约束，状态机还会阻止待审批退款对应工单被错误关闭。

## 3:10～4:20 评测与回归

操作：打开 Evaluation Workbench 和真实模型报告。

展示重点：

- 18 条 Case 与6类能力分组。
- 断言不仅检查最终文本，也检查工具轨迹、审批和最终业务状态。
- 展示 `recovered` / `regressed`。
- 展示 DeepSeek 两轮全量都为14/18，但失败集合发生漂移。

讲解：

> 连续真实模型评测发现，单次修复可能让旧 Case 恢复，同时产生新的回归。因此我把工具选择、审批属性、规则关键词和写入资格继续下沉到确定性执行层，而不是只调整 Prompt。

## 4:20～4:50 收尾

> 当前版本是单机工程沙箱，已经覆盖执行、审批、Trace、恢复和评测闭环。生产化下一步会把本地 JSON 替换为 PostgreSQL，并使用任务队列、幂等键和事务边界支持分布式执行恢复。

## 录制验收清单

- 视频控制在3～5分钟，避免逐行讲代码。
- 至少展示一次审批拒绝后的零副作用。
- 明确区分 Mock 18/18 与真实模型全量14/18。
- 不展示密钥、私人路径或无关桌面内容。
- 结尾给出仓库地址、启动命令和架构文档位置。
