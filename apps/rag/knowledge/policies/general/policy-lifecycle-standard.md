# 企业政策发布、版本与失效管理标准

## 一、适用范围

本标准适用于进入企业政策知识库的正式制度、操作指引、审批矩阵、FAQ 和历史版本。它规定文档从草拟、审核、发布、索引、修订到失效的完整生命周期，保证 Agent 检索到的内容具有明确权威性。临时聊天记录、未经确认的会议纪要和个人笔记不得直接作为 active 政策发布。

## 二、文档身份

每项政策必须具有稳定 policy_id。修订同一政策时沿用 policy_id 并提升 version；改变业务领域或形成独立审批依据时创建新的 policy_id。标题可以为了清晰而调整，但不能用标题替代稳定标识。来源文件名应可读且不包含密钥、客户信息或个人姓名。

version 使用可比较的主次版本格式。改变适用条件、审批阈值、禁止事项或责任部门时提升主版本；仅澄清措辞、补充示例且不改变业务结果时提升次版本。effective_date 表示政策开始适用于业务判断的日期，不等于文件上传日期。

## 三、状态

active 表示当前可以作为业务依据。archived 表示仅供历史审计，默认检索必须排除。未来可以扩展 draft 和 scheduled，但首期知识库不导入未审批草稿。任何状态变化都需要记录负责人、时间和原因。

同一 policy_id 可以保留多个历史版本，但在同一适用日期范围内只能有一个默认 active 版本。新版本生效时，旧版本应原子地转为 archived，避免短时间同时命中两套规则。索引更新失败时继续使用上一版，不得出现没有可用政策的半完成状态。

## 四、发布审核

政策内容由业务部门编写，风险与合规人员审核适用范围、冲突和审批边界，知识库管理员审核元数据、格式和引用。涉及资金、合同、安全和个人信息的政策必须经过对应专业负责人确认。审核不能由 Agent 自动完成。

发布前需要运行 Schema 校验、链接检查、重复 ID 检查、active 版本检查和检索回归。新增政策必须增加正例、难负例和无答案评测；修改政策应更新受影响的期望答案，并解释行为变化。

## 五、内容结构

正式政策至少包含目的、适用范围、定义、受理条件、禁止条件、审批或责任、操作记录和例外。FAQ 可以使用问答结构，但必须引用其解释的正式政策，不能创建新的宽松权限。操作指引可以描述系统步骤，但不能改变 Tool Registry 风险等级。

每个章节应能够独立理解，同时保留文档标题和版本元数据。模糊词语如“及时”“较大金额”“特殊客户”应给出具体口径或责任人。无法量化的判断必须明确要求人工确认，不能留给模型自由解释。

## 六、冲突与优先级审查

发布人应搜索相同关键词和相邻业务领域，检查新政策是否与现有内容冲突。退款政策与审批矩阵、订单取消与退款、SLA 补偿与现金退款、合同升级与续费折扣是重点交叉领域。

专项业务规则规定是否满足业务条件，通用控制规则规定执行方式，两者通常共同适用而非互相替代。例如退款政策允许申请，并不取消所有退款必须人工审批的控制。如果确实需要例外，政策必须明确例外范围、批准角色和审计要求。

出现无法消除的冲突时不得发布 active 版本。已经在线的冲突被发现后，应标记知识库告警并暂停相关自动写入，直到政策负责人给出书面解释。

## 七、索引与校验和

RAG 服务对规范化后的文档内容计算校验和。文件内容和关键元数据均未变化时跳过重新 Embedding。修改 status、version、effective_date 或正文均视为新索引输入。分块参数和 Embedding 模型版本改变时，即使原文不变也需要重建索引，并记录新的索引版本。

导入过程先解析并验证整个文档，再生成新 Node 和向量。所有新数据成功写入后，才切换文档当前索引指针并删除旧 Node。任何解析、Embedding 或数据库错误都必须使本次导入失败，不得留下部分 Node。

## 八、历史版本检索

普通搜索只检索 active 且已经生效的政策。用户明确询问历史日期、旧规则或审计原因时，可以开启 include_archived，并把查询日期传给检索层。结果必须同时显示 archived 状态和适用日期，不得把历史内容混入当前操作建议。

历史 Trace 保存当时返回的政策 ID、版本、Node ID 和正文片段。即使源文件后来删除或归档，已完成 Trace 仍保留引用快照。删除上传文件前应确认满足审计保留要求。

## 九、紧急修订

安全事件或重大合规变化可能要求紧急发布。紧急流程可以缩短评审时间，但仍必须有业务负责人和风险负责人双人确认，仍必须通过 Schema、版本和基本检索测试。紧急政策应标记复核日期，并在事件结束后完成正式回顾。

不得通过直接修改数据库 Node、手工覆盖向量或在 Prompt 中增加临时规则替代政策发布。临时控制应放在服务端可审计配置中，并尽快形成正式政策。

## 十、监控与回滚

知识库应监控文档数、active 版本数、索引失败数、无匹配率和旧版本命中情况。新版本发布后，如果核心 Golden Query Recall@5 或 Top-1 显著下降，应回滚到上一索引版本并调查分块、内容或元数据变化。

回滚只改变当前可检索索引，不删除失败版本和审计记录。回滚完成后重新运行完整检索评测，并记录影响范围、开始与恢复时间。

## 十一、职责分离

文档作者不能独自完成发布和删除；知识库管理员不能自行改变业务含义；开发人员不能为了提高评测分数修改政策事实。所有变更应经过代码评审或等价审批，并能够关联到具体提交、发布记录和评测报告。

## 十二、示例

VIP 退款政策从 1.0 升级至 2.0 时，新增了所有退款必须通过人工审批的要求。这是业务控制变化，应提升主版本，把 1.0 标记 archived，并新增“快速退款是否免审批”的难负例。普通查询只能返回 2.0；询问 2025 年旧订单时可以同时返回 1.0，但必须说明其已失效。

发票 FAQ 仅补充抬头更正材料，没有改变正式发票政策，因此可以作为独立 FAQ policy_id 发布。它不能因为客户不愿补材料而建议退款，也不能覆盖正式政策中的付款主体核对要求。
