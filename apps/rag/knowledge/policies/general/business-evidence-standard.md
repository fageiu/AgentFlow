# Agent 业务证据采集与引用标准

## 一、目的

本标准规定 Agent 在工单处理、政策检索、退款申请、人工审批和最终答复中如何采集、验证、引用和保存业务证据。目标是让每一项业务判断能够追溯到真实工具结果或当前有效政策，避免把模型推测、用户陈述和系统事实混为一谈。本标准适用于所有自动化运行，不因模型、Prompt 或客户端不同而改变。

## 二、证据等级

一级证据是由受控业务工具直接读取或写入后返回的结构化结果，包括工单、客户、订单、退款记录和审批状态。一级证据可以证明对象在某一时刻的真实状态，但调用失败、超时或被取消时不得沿用上一次结果。工具返回中的 operation 字段用于区分 created、reused、updated 和 unchanged；只有 created 或 updated 可以证明本次运行产生了新副作用。

二级证据是知识库返回的当前有效政策 Node。每条引用必须包含政策编号、版本、来源文件、章节或页码。政策证据可以说明允许、禁止和审批条件，但不能证明某个订单实际满足条件；判断需要同时结合一级业务事实。archived 政策只允许解释历史决策，不能作为当前自动操作依据。

三级证据是用户在当前任务中提供的描述、附件或补充说明。用户陈述可以触发核查，但不能直接证明客户等级、订单金额、退款状态或审批结果。存在冲突时以受控工具和权威系统记录为准，并在结论中指出差异。

模型生成的计划、推理和自然语言不属于业务证据。它们可以帮助组织任务，但不能证明工具已经执行、审批已经通过或政策确实命中。模型引用不存在的 evidence ID、声称未发生的写入或改变可信 decision 时，服务端必须拒绝该结果并回退确定性 Outcome。

## 三、最小证据集合

退款判断至少需要工单、客户等级、订单金额与状态、完成日期、既有退款状态、当前退款政策和审批矩阵。缺少任一关键事实时只能给出待核查结论，不能创建退款。订单取消咨询至少需要订单状态、付款状态和交付事实；客户明确要求返款时再补充退款政策。发票咨询至少需要订单和开票主体，不得因为缺少税务材料转而执行退款。

SLA 判断至少需要事件起止时间、影响范围、服务责任、客户合同等级和排除条款。监控数据未确认时，客户描述只能作为待验证线索。合同升级和续费折扣争议至少需要当前合同版本、有效期、订单金额、报价或活动证据；没有正式依据时不得生成有法律约束力的报价。

高风险工单关闭至少需要原诉求处理结果、未完成审批状态、工单优先级、客户风险情况和人工负责人决议。即使其他业务步骤已经完成，也不能用“看起来解决了”替代关闭审批。

## 四、检索证据要求

知识库查询应使用用户目标和可信工单上下文组成自然语言问题。keyword_hint 只能用于排序提示，不能强制过滤到某个领域。检索系统应返回多个候选、相关度、版本和引用位置；应用层只能引用实际返回的 Node，不得根据政策标题补写未检索到的内容。

当多个结果来自同一政策的相邻段落时，可以合并展示，但仍应保留各 Node ID。多个政策同时适用时，按照专项规则与通用控制规则共同满足的原则组织依据。结果包含历史版本时，必须明确其 archived 状态并优先寻找当前版本。

相关度低于系统阈值、候选相互冲突或没有覆盖用户问题时，应返回无可靠匹配。无匹配不是系统错误，Agent 可以改写一次查询；再次失败后应明确说明知识库没有足够依据，不得用模型常识补齐政策。

## 五、引用格式

面向用户的处理依据使用“政策标题（政策编号，版本，章节或页码）”格式。内部 EvidencePacket 还应保存 Document ID、Node ID、来源文件、检索分数和查询关联的 Run ID。引用不得只写“根据公司规定”或“系统显示”，因为这种表述无法审计。

对业务事实的引用应说明对象和关键字段，例如“订单 O-7001 金额 6800 元，状态 completed”。涉及时间判断时同时记录业务基准日期和来源日期。涉及写入时说明工具实际 operation，而不是只说明调用过工具。

## 六、冲突处理

用户陈述与工具事实冲突时，先向用户说明系统记录，并允许其补充材料；不得直接改写系统状态。两个当前有效政策冲突时，先检查适用范围、发布日期和专项优先级。仍无法解释时停止自动写入并转合规人员，不得选择更宽松或更有利于完成任务的规则。

工具结果前后不一致可能表示并发修改。Executor 应重新读取关键对象或停止运行，不能把较早状态继续传给写工具。审批结果与业务状态不一致时，以服务端审批记录和实际写入轨迹分别陈述，禁止推断未发生的补偿动作。

## 七、隐私与保留

Trace 只保存完成审计所需的字段，不保存 API Key、访问令牌、完整银行卡或无关个人信息。政策引用可以保存正文片段，但应限制长度，避免把整个受限文档复制到对话历史。日志可以记录 query hash、Node ID 和耗时，不应默认记录完整用户查询和政策正文。

运行历史、审批记录和评测结果的保留周期由部署环境决定。删除知识文档不会修改已经完成的历史 Trace；历史 Trace 应继续显示当时引用的政策版本，以保证审计一致性。

## 八、评测要求

检索评测需要覆盖准确问法、同义表达、相似政策、错误 keyword_hint、历史版本和无答案问题。Agent 评测除检查最终回答外，还必须检查真实工具轨迹、引用政策、审批行为和最终业务副作用。只要真实 Provider 发生 Mock fallback，该用例不能作为真实质量通过。

评测失败应保留 query、预期政策、实际 Top-K、各阶段分数和 Node 引用。优化时先区分解析、分块、Embedding、召回、融合、重排和阈值问题，禁止只修改最终 Prompt 掩盖检索缺陷。

## 九、职责

业务政策负责人维护政策内容和版本；数据治理人员维护元数据和访问范围；RAG 服务负责人保证检索、引用和索引一致性；Agent Runtime 负责人保证工具授权、错误处理和可信 Outcome；评测负责人维护独立 Golden Query。任何角色都不能通过修改单一层绕过其他控制。

## 十、示例

客户提出“接口停了两小时，帮我把费用退掉”。正确流程是先读取工单、客户和订单，检索 SLA 事件政策与补偿指引，确认是否为平台责任，再根据客户明确退款诉求检索退款政策和审批矩阵。SLA 文档可以证明需要核查和升级，但不能证明事件真实持续两小时；工单描述可以说明客户主张，但不能替代监控记录。只有全部退款条件满足且人工审批通过后，写工具才能执行。

客户提出“发票抬头写错了，上一轮被拒绝，重新处理”。正确流程是检索发票更正问答并读取拒绝原因，要求补充缺失材料或转财务复核。即使订单金额较大，也没有退款诉求和退款政策依据，因此不得创建退款或进入资金审批。
