# Agent 客户沟通与业务结论表达规范

## 基本要求

Agent 回复应区分客户诉求、已经核实的事实、适用政策、实际执行动作和下一步责任人。语气可以自然，但不能用确定语气描述尚未验证或尚未发生的动作。所有高风险结论以服务端 Outcome 为准，模型只负责在可信事实范围内组织语言。

标准结论包含“工单需求、处理结果、处理依据、下一步”四部分。工单需求来自用户任务和真实工单描述；处理结果说明是否只读、无需退款、需要审批、已经满足、人工拒绝、失败或取消；处理依据引用业务事实和政策；下一步明确由 Agent、人工或客服负责。

## 不得虚构动作

如果工具没有返回 created 或 updated，不得写“已经创建”“已经更新”。operation 为 reused 时说明已有记录被复用；unchanged 时说明状态本来已经满足。审批等待时写“等待人工审批”，不能提前写“退款已办理”。审批拒绝时明确没有产生退款和后续状态写入。

模型建议不能改变 decision。即使模型认为客户应该退款，只要可信轨迹不支持，Outcome 仍为 no_refund 或 manual_review。模型引用未知 evidence ID 时使用确定性结论，不把无依据推理展示给用户。

## 政策引用

处理依据使用政策标题、编号、版本和章节或页码。不能只写“按照规定”。当多个政策共同适用时分别说明其作用，例如退款政策判断受理条件，审批矩阵判断批准角色。历史版本只用于历史问题，并标记已经失效。

引用正文应保持必要上下文，不截取造成相反含义。知识库没有可靠匹配时明确说明缺少政策依据，并建议人工确认，不使用通用常识编造企业规则。

## 退款沟通

符合退款受理条件不等于已经退款。回复先说明订单与客户事实，再说明需要的审批层级和当前状态。金额、到账时间和处理时限没有真实依据时不作承诺。重复退款应说明复用现有申请和当前进度。

退款被拒绝时可以说明审批原因和补充材料路径，但不能暗示重新提交必然通过。用户只是咨询发票、SLA、升级或取消时，不应主动引导退款，除非用户明确提出且政策要求进一步核查。

## 发票沟通

发票咨询说明需要核对的订单、付款主体和材料。抬头更正被拒绝时指出缺失材料或主体不一致，不把它描述为资金退款。涉及税务信息时避免在公开回复中重复完整税号或地址。

## SLA 沟通

区分首次响应、恢复时间和可用率。客户报告的中断时长尚未通过监控确认时使用“客户反馈”而非“已确认故障”。补偿只说明评估流程和责任团队，不自行计算现金金额。

## 合同与订单沟通

升级和续费报价由商务确认，Agent 可以说明所需信息和流程，不生成最终报价。订单取消时区分未付款、已付款、未交付和已经产生定制成本，避免“一取消就免费”或“一取消就退款”的绝对表述。

## 错误与取消

失败回复说明任务未完成、是否产生副作用和可执行建议。不要向用户展示堆栈、内部 URL、密钥或数据库细节。用户取消时说明后续步骤已经停止，并建议确认现有业务状态后重试。

知识服务不可用时说明暂时无法核实政策，不返回 Seed Fixture 内容。对象不存在时建议核对 ID，不列出其他客户数据。持久化失败时说明结果无法可靠保存，高风险写入不得继续。

## 敏感信息

回复遵循最小披露。客户只能看到与自身工单相关的事实。风险分、内部审批人名单、系统配置和安全检测细节不应完整暴露。Trace 面向开发者展示时仍需对 Token 和个人信息脱敏。

## 示例一：退款待审批

工单需求：VIP 客户申请订单退款。处理结果：已经完成客户、订单和政策核查，退款操作尚未执行，当前等待人工审批。处理依据：订单金额和完成日期来自订单工具，受理条件来自 VIP 客户退款管理办法 2.0，审批层级来自退款审批权限矩阵 1.2。下一步：由客户成功主管和财务经理完成审批，批准后系统才执行退款工具。

## 示例二：发票更正

工单需求：客户补开发票时发现抬头错误。处理结果：属于发票更正咨询，不支持据此创建退款。处理依据：订单已读取，发票抬头更正操作问答要求核对原发票、正确抬头和拒绝原因。下一步：客服收集缺失材料，付款主体不一致时转财务复核。

## 示例三：无政策匹配

工单需求：用户询问与当前企业政策库无关的问题。处理结果：知识库没有达到可信阈值的政策证据，未执行任何业务写入。处理依据：两次检索均返回 KNOWLEDGE_NO_MATCH。下一步：请提供与工单相关的具体业务目标，或由人工确认是否需要新增政策。

## 质量检查

自动检查固定字段是否完整、政策引用是否来自真实 Match、动作是否存在可信 Tool Result、责任人是否与推荐一致。评测自然语言只断言稳定实体和关键禁止项，不用固定措辞替代业务语义。
