# Agent 异常、重试与安全失败处理标准

## 基本原则

Agent 遇到数据不存在、工具异常、模型异常、知识库不可用、审批中断或系统故障时，应优先保持业务状态安全和可解释。失败不能通过模型补写一段看似成功的回复掩盖，也不能为了“完成任务”调用未授权工具。所有异常必须转换为稳定错误码、用户提示、重试属性和 Trace 上下文。

## 错误分类

业务错误表示请求对象不存在、状态不满足或政策没有可靠匹配。工具错误表示参数校验、外部 API、数据库或业务函数失败。模型错误包括超时、限流、服务端错误和响应格式错误。系统错误包括持久化目录、进程资源和未知编程异常。分类决定重试和用户建议，但不改变工具风险等级。

知识服务不可用使用 KNOWLEDGE_SERVICE_UNAVAILABLE；模型或索引尚未就绪使用 KNOWLEDGE_INDEX_NOT_READY；没有达到阈值的政策使用 KNOWLEDGE_NO_MATCH；文档解析和元数据错误使用 KNOWLEDGE_DOCUMENT_INVALID。无匹配不等于可以用 Seed Policy 或模型常识回退。

## 重试条件

只有明确标记 retryable 的错误才自动重试。网络瞬时错误、429、部分 5xx 和短暂数据库连接失败可以有限重试；Schema 错误、对象不存在、审批拒绝和文档格式错误不应原样重试。每次重试必须有上限、退避和取消信号。

知识查询无匹配时允许 Agent 改写一次自然语言 query，例如补充可信工单描述，但不得只反复替换关键词。第二次仍无可靠结果时结束知识步骤并说明缺少依据。工具参数校验失败可以让模型修正一次，重复失败进入结构化错误。

## 取消

Run 控制器在模型请求、重试等待和步骤边界检查取消信号。用户取消后不得触发 Mock fallback、不得继续等待审批、不得执行下一写步骤。已经开始的不可取消外部写入需要通过幂等查询确认结果，不能盲目补偿。

取消不是失败，不应标记为可自动重试成功。Outcome 使用 cancelled decision，并提示用户确认业务状态后重新发起。取消请求本身保持幂等。

## 部分成功与补偿

退款创建和工单状态同步属于同一业务单元。退款创建成功但状态更新失败时，应使用定向补偿恢复该订单、工单和退款记录，不能重置整个沙箱影响其他 Run。补偿成功与否都要写入 Trace。

生产数据库应使用事务、幂等键或 Saga/Outbox 保障一致性。补偿逻辑不能删除其他并发 Run 已完成的结果。无法判断所有权时停止自动修改并转人工。

## Provider 降级

没有 API Key 时可以显式使用 Mock 保证本地 Demo 可运行。真实 Provider 请求失败时，只有配置允许才使用 Mock fallback，并把 provider、model 和原因写入结构化元数据。真实模型评测中禁止 fallback，任何触发都不得判定通过。

知识检索真实模式不得静默使用内存政策回退。测试 Fixture 必须通过 RAG_MODE=fixture 显式启用，并在 Trace 或配置中可辨认。

## 数据不存在

工单、客户或订单不存在时，应立即停止依赖该对象的后续步骤，禁止进入审批或调用写工具。错误提示可以建议核对 ID，但不能列出其他客户对象。小写或常见格式差异可以在服务端规范化，规范化结果应写入真实执行上下文。

政策无匹配时不应把它伪装为 Policy not found 后反复调用固定关键词。系统应保存 query、hint、候选分数和阈值，帮助判断是语料缺失还是检索质量问题。

## 持久化失败

如果 Run Trace 或关键状态无法持久化，系统应进入降级并报告 STORAGE_WRITE_ERROR。高风险写入前无法保证审计持久化时应停止业务写入。只读查询可以根据产品策略返回临时结果，但必须明确不可恢复和不可审计。

磁盘权限、空间和文件占用属于运维问题，不应让模型尝试修改路径。生产版本迁移数据库后仍需处理连接池耗尽、事务冲突和只读副本延迟。

## 安全输出

用户提示说明发生了什么、是否产生副作用和下一步，不暴露内部堆栈、密钥、完整 URL 参数或其他客户数据。详细诊断保存在服务端 Trace，并对前端展示做长度限制和脱敏。

未知异常统一归一化，但不能丢失 phase、toolName、toolCallId 和 runId 等执行上下文。错误文案变化不应改变稳定 code。

## 可观测指标

每类错误统计数量、重试次数、最终状态和恢复率。模型错误记录 provider、model、耗时和 fallback；知识错误记录索引状态、候选数和阈值；工具错误记录工具名和是否发生副作用。指标不应以原始用户文本作为高基数标签。

## 评测场景

必须覆盖不存在工单、非法工具参数、模型非法 JSON、Provider 5xx 后恢复、超时 fallback、工具循环超限、知识服务 503、索引未就绪、无答案拒答、持久化不可写、退款后续失败补偿和取消审批等待。

每个失败 Case 都应断言禁止的写工具、最终状态、错误码、退款记录数和工单状态。只检查返回 500 不能证明业务安全。
