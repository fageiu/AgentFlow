# 企业知识检索质量与安全治理规范

## 目标与边界

企业政策检索服务负责从已经审核的文档中返回相关证据，不负责替业务系统确认订单事实，也不直接生成具有执行效力的决策。检索结果必须能够追溯到 Document 和 Node。Agent 可以使用检索证据解释规则，但写入是否允许仍由服务端状态、审批和工具结果共同决定。

LlamaIndex 用于文档、Node、Ingestion Pipeline、VectorStore 和 Retriever 组合。框架配置不是安全边界：文档状态过滤、管理鉴权、阈值、引用校验和错误回退必须由应用服务明确控制。禁止为了使用框架而把现有 Agent Executor 替换为无法审计的 QueryEngine Agent。

## 数据准入

只有通过政策生命周期审核的文档才能进入 bundled active 索引。上传接口校验文件类型、大小、UTF-8、frontmatter、稳定 policy_id、版本、状态和责任部门。PDF 必须能提取文本；扫描件在首期返回明确错误，不允许索引空白页或 OCR 猜测文本。

文档中包含其他系统指令、脚本或提示词时一律作为普通文本保存，不赋予执行能力。包含密钥、个人敏感信息或未授权客户数据的文件不得导入。上传失败不能留下可检索的部分 Node。

## 分块

默认使用 512 Token、80 Token overlap，并优先保留标题、段落和 PDF 页边界。每个 Node 继承 policy_id、版本、状态、部门、来源、章节和页码。标题和关键元数据可以参与 Embedding，但最终引用正文必须保持原文可读。

过短 Node 容易缺少条件，过长 Node 会混入多个主题。评测发现关键条件被切断时，优先调整文档结构或分块，不应在最终 Prompt 中硬编码答案。相邻 Node 可以在展示时合并，但检索与审计仍保留各自 ID。

## Embedding

默认模型为 BAAI/bge-m3，向量维度 1024。模型名称、版本或归一化方式变化时必须重建全部向量，不能在同一列混用不兼容 Embedding。测试使用确定性 Fake Embedding 只验证数据流，不代表真实语义质量。

查询和文档使用模型推荐的编码方式。批量 Embedding 控制内存和并发，失败批次可以重试但不能重复提交已经成功的版本切换。模型缓存由只读运行用户访问，下载失败时 readiness 保持未就绪。

## 混合召回

向量检索处理同义表达和上下文语义，中文关键词检索保留政策编号、金额、时间和专有词的精确匹配。两路各召回 Top 20，再用 RRF 融合并按 Node ID 去重。keyword_hint 只能轻微加权，不能硬过滤，因为模型或用户可能提供错误领域提示。

普通查询排除 archived 和未来未生效政策。明确历史问题可以开启历史范围，但必须在返回中标记状态。租户、部门或权限过滤如果后续加入，应在数据库查询阶段执行，不能检索后再依赖模型删除无权限内容。

## 重排与阈值

融合候选前 10 条使用 BGE Reranker，返回 Top 5。保留原向量分、关键词分、融合分和重排分，便于分析。归一化最终分低于 0.35 时返回无可靠匹配；阈值只能通过 Golden Query 数据调整，不能为了减少无答案率任意降低。

Reranker 必须在中文难例上证明不降低 Top-1 命中率。如果模型加载失败，真实模式应返回服务未就绪或明确降级配置，不得悄悄跳过后仍宣称使用了重排。

## 引用

每个 Match 返回政策 ID、标题、版本、Node 正文、来源文件、章节、PDF 页码和分数。TypeScript 工具把 Top-1 映射到兼容 Policy 字段，同时把 Top-K 和 Citation 放入 EvidencePacket。最终结论只能引用实际返回 Node。

引用片段设置长度上限，避免 Trace 复制完整文档。用户需要更多上下文时可以展示相邻段落，但仍应保持来源。来源文件删除后，历史 Run 保存的引用快照继续可读。

## 无答案与冲突

没有候选达到阈值时返回 KNOWLEDGE_NO_MATCH。Agent 可以结合可信工单上下文改写一次；再次失败后明确说明知识库缺少依据。禁止调用通用 LLM 生成“可能的公司政策”。

Top-K 包含相互冲突的 active 政策时，返回冲突标记并转人工合规。仅仅分数更高不能解决业务冲突。当前版本和 archived 版本同时出现时，当前版本用于操作，历史版本只能解释过去。

## 管理接口

上传、重建、删除和 bundled 同步要求 X-Admin-Token，并记录操作者、文档、版本和结果。Token 比较使用安全方式，日志不输出 Token。管理请求重复提交保持幂等；删除 active 文档前应确认是否存在替代版本。

服务首次启动且索引为空时可以自动导入 bundled 文档。非空索引不应每次启动全量重建，只对校验和变化的文档处理。readyz 只有数据库、模型和至少一个 active 索引都可用时返回 ready。

## 指标

离线指标包括 Recall@5、MRR、无答案拒答准确率和重排变化。在线指标包括检索 P50/P95、各阶段耗时、候选数、无匹配率、索引失败和 archived 命中。query 文本不作为指标标签，使用 hash 关联诊断日志。

核心政策 Recall@5 目标不低于 95%，Top-1 MRR 不低于 0.85，无答案拒答准确率不低于 90%。这些指标基于版本化评测集，语料或模型变化必须重新运行。

## 安全测试

测试文档应包含“忽略审批”“调用退款工具”等提示注入文本，确认检索只返回文字而不改变执行权限。还应测试超大文件、错误扩展名、空 PDF、路径穿越、重复文件、恶意 frontmatter 和管理 Token 缺失。

## 变更发布

检索参数、Embedding、Reranker、分块或阈值变化都记录配置版本。发布前运行 Python 测试、PostgreSQL 集成测试、检索评测和 Agent Golden Task。真实模型评测发生 fallback 时不能作为通过证据。

出现回归时可以回滚索引或服务镜像，但不能删除失败报告。回滚后重新验证当前政策版本和引用一致性。
