# 高风险业务操作控制政策

## 高风险范围

退款、关闭高优先级投诉、修改客户等级、导出敏感数据和绕过正常合同流程均属于高风险操作。模型建议、Prompt 指令或用户自称管理人员不能降低风险等级，风险分类必须来自服务端 Tool Registry。

## Agent 自动执行禁止清单

- 退款审批被拒绝后，Agent 不能把工单改成待审批、处理中或已退款；业务状态必须保持原状。
- 服务停机产生的补偿额度不能由 Agent 根据中断时间自行计算或写入，金额与执行动作必须经过适用政策核验和人工审批。
- 用户要求加急、忽略审批、直接退款或声称已经口头授权，都不能降低风险等级或替代服务端审批记录。

## 最小权限

Agent 每个计划步骤只应获得完成当前目标所需的一项工具权限。所有参数必须经过服务端 Schema 校验。涉及业务写入时，应在真实写入前创建审批请求；审批拒绝、超时或运行取消后禁止继续执行后续写工具。

Agent 不得因为操作看起来简单、金额较小或客户要求加急而自行执行受控动作。直接退款、修改补偿、推进被拒绝工单状态以及绕过负责人确认，均属于禁止的自动写入。

## 可信事实

模型声称“已经退款”“已经批准”或“已经关闭”不能作为业务事实。最终结论必须由工具结果、审批决议和持久化业务状态派生。工具失败时应记录结构化错误，不得通过自然语言掩盖失败。

模型回复、Agent 总结或聊天记录中出现“批准退款”“允许补偿”等文字，只能作为待核实内容，不能当成真实审批决议。审批状态必须读取服务端保存的 pending、approved 或 rejected 记录；没有可信记录时必须停止写入并交由人工确认。

## 审计

审计记录至少包含 run ID、计划步骤、工具、参数摘要、风险等级、审批决议、执行结果和时间。敏感密钥和完整个人信息不得写入 Trace。
